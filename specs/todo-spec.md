# TODO 管理規範

適用情境：任何由 AI 工具協作的軟體專案，不限語言或平台。

---

## 核心原則

**`TODO.md` 是唯一的待辦入口，只列未完成項目。**

細節與結案紀錄留在各自的來源文件（設計文件、部署指南、架構紀錄等），`TODO.md` 只負責「今天有什麼要做」，不保存歷史。

---

## 檔案結構

`TODO.md` 放在專案根目錄，格式如下：

```markdown
# <專案名稱> — 待辦事項

> 僅列未完成項目。結案時從此處刪除該行，並在對應文件標記 `[x]`。
> 最後更新：YYYY-MM-DD

---

## <分類一>

- [ ] **項目名稱**：一行說明。→ [`來源文件.md`](./路徑/來源文件.md)
  - [ ] 子項目一
  - [ ] 子項目二

## <分類二>

- [ ] **項目名稱**：一行說明。→ [`來源文件.md`](./路徑/來源文件.md)
```

### 格式規則

- 父項格式固定：`- [ ] **名稱**：說明。→ 連結`
- 父項可拆成子項，子項格式固定：`  - [ ] 子項目`
- 說明以「行動方向」為主，不寫已知結論（那是來源文件的工作）
- 父項連結指向能找到完整背景的那份文件，而非通用首頁
- 子項只列可驗證的細分動作，不重複放來源連結
- 分類名稱依專案性質自訂，常見分類：功能開發、監控補齊、維運強化、技術債、文件

---

## 維護流程

### 新增待辦

1. 在 `TODO.md` 對應分類下加一行（格式如上）
2. 在來源文件中補充細節（背景、限制、嘗試過什麼）

兩個步驟都要做。只更新 `TODO.md` 而沒有來源文件的背景，等同於留下一個沒有上下文的任務。

### 結案

1. 從 `TODO.md` **刪除**該行
2. 在來源文件將對應項目標記為 `[x]`，並補上結案說明（何時、怎麼確認、結論是什麼）
3. 更新 `TODO.md` 頂部的「最後更新」日期

兩個步驟都要做。只從 `TODO.md` 刪掉而沒有在來源文件留下紀錄，等同於讓結案原因消失。

## AI 行為協定

### 每次對話開始時

若使用者詢問「專案待辦」、「目前進度」、「還有什麼要做」，AI 應：

1. 讀取 `TODO.md`
2. 直接列出目前未完成項目，不重新翻找各文件

### 結案時

使用者確認結案後，AI 應依序：

1. 從 `TODO.md` 刪除該行
2. 在來源文件標記 `[x]` 並補上結案說明
3. 更新「最後更新」日期
4. 列出剩餘待辦清單

不得只做其中一步。

### 新增待辦時

1. 先問清楚：放哪個分類？對應哪份文件？
2. 同步更新 `TODO.md` 與來源文件

### 查詢待辦時

直接從 `TODO.md` 回答，不需要重新掃全專案。如果使用者要了解某個項目的背景，才跟著連結去讀來源文件。

---

## 禁止的行為

```
# 結案時只改 TODO.md，沒有在來源文件標 [x]               ← 禁止
# 新增待辦時只改 TODO.md，沒有在來源文件補細節            ← 禁止
# TODO.md 保留已完成項目（歷史紀錄應在來源文件）          ← 禁止
# 在 TODO.md 中寫大量細節（細節屬於來源文件）             ← 禁止
# AI 在沒有讀 TODO.md 的情況下回答「目前待辦是什麼」       ← 禁止
# 長期不更新「最後更新」日期                               ← 禁止
```

---

## 套用流程

**在新專案導入：**

1. 將此檔案複製至專案根目錄，命名為 `todo-spec.md`
2. 在專案根目錄建立 `TODO.md`，填入初始待辦項目
3. 確認每個項目都有對應的來源文件連結
4. 在 `CLAUDE.md`（或對應 AI 工具的設定檔）加入：

```
請遵照 todo-spec.md 的規範管理 TODO.md。
```

**與 AI 工具協作時告知：**

> 「請遵照 `todo-spec.md` 的規範，結案時同步更新 `TODO.md` 與來源文件，並在列出待辦時直接讀取 `TODO.md`。」

## 使用方式

| 工具 | 整合方式 |
|------|----------|
| Claude Code | 在 `CLAUDE.md` 加入 `參考 todo-spec.md 的 TODO 管理規範` |
| Cursor | 在 `.cursorrules` 加入相同指示 |
| 其他工具 | 對話開頭直接引用本檔案內容 |
