# 中文註解風格規範

適用語言：Shell、Python、Makefile、YAML、任何腳本或部署相關檔案。

---

## 核心原則

**說明決策與風險，不逐行翻譯指令。**

好的註解讓讀者知道「為什麼這樣做」，而不是把指令用中文複述一遍。

---

## 區段標題

用中文區段標題分隔主要流程，讓檔案結構一目瞭然。
依使用位置採用不同格式：

### 模組層級（檔案開頭、類別前）

用 `# ===...===` 大框標示，視覺區隔最強，適合分隔獨立的功能區塊：

```python
# ==========================================
# 1. 環境設定
# ==========================================
# 說明......
```

### 函式內步驟

用 `# ── 標題 ──` 短橫線標示，輕量一行，適合標記函式內的執行階段：

```python
# ── 1. 載入 env ──
# ── 2. 過濾卡片 ──
```

常見區段範例（視專案調整）：
- 基本路徑與環境設定
- 環境變數載入與前置條件檢查
- 共用工具函式
- 資料庫部署與健康檢查
- 應用服務部署與健康檢查
- 主流程

---

## 函式與區段的說明格式

每個主要函式或區段前，應說明以下三點：

1. **目的**：這段程式碼做什麼。
2. **原因**：為什麼這樣設計，有什麼限制或決策背景。
3. **敏感資料標注**：若涉及密碼、token、DB URL、資料寫入，需明確標注，提醒維護者注意外洩風險。

範例（Shell）：
```sh
# 載入 .env 並驗證必要變數
# 原因：CI 環境與本機環境的變數來源不同，統一在這裡處理避免漏掉
# 注意：此區段會讀取 DB_PASSWORD 與 API_TOKEN，請勿將這段輸出 print 至 log
```

範例（Python）：
```python
# 建立資料庫連線池
# 原因：單次連線在高並發下會超時，連線池可重用連線降低延遲
# 注意：DB_URL 含帳號密碼，禁止印出此變數內容
```

## 環境限制標注

若程式碼有特定環境依賴，應在相關段落說明原因，避免後人誤刪或重複踩坑。

需標注的常見情境：
- 公司內網 / Proxy 設定
- WSL 與原生 Linux 的路徑差異
- Docker 自訂網路名稱的原因
- 首次執行需要手動登入的流程
- 僅在特定 CI/CD 平台有效的環境變數

範例：
```sh
# 以下 proxy 設定僅在公司內網環境需要
# 原因：公司防火牆封鎖直連 Docker Hub，需透過內部 proxy 才能拉取 image
export HTTPS_PROXY=http://proxy.internal:3128
```

## 適用範圍

**所有可執行程式，不論長短，一律補齊。** 不可因「腳本太簡單」而跳過。

最低要求：
- 檔案開頭有一行說明用途的標頭註解
- `main()` 或主流程中有區段步驟標示

---

## 套用流程

每次套用本規範前，**必須先列出所有目標檔案**，再逐一處理，不得邊做邊跳過。

1. 用 `find` 或 `ls` 列出目標目錄下所有程式檔案
2. 將清單完整列出，每個檔案對應一個待處理項目
3. 逐一檢查並補齊，處理完才能標記為完成
4. 全部完成後回報，不得在未處理的情況下聲稱「已完成」

**禁止的行為：**
- 掃描到「看起來很簡單」的檔案就跳過
- 未列清單就直接開始編輯
- 編輯幾個檔案後自行認定「其餘應該沒問題」

---

## 禁止的寫法

```sh
# 執行 docker compose up        ← 沒有說明為什麼，沒有價值
# 設定變數 DB_HOST              ← 逐行翻譯指令，無意義
```

## 使用方式

將此檔案複製至專案根目錄，並在與 AI 工具協作時告知：

> 「請遵照 `comment-style.md` 的風格撰寫中文註解。」

各工具整合方式：
- **Claude Code**：在 `CLAUDE.md` 加入 `參考 comment-style.md 的中文註解風格規範`
- **Cursor**：在 `.cursorrules` 加入相同指示
- **其他工具**：對話開頭直接引用本檔案內容
