# KeePassXC 備份規範

適用情境：任何含有敏感設定、密鑰或不進 git 的檔案的專案。

---

## 核心原則

**`.gitignore` 排除的敏感檔案，必須在 KeePassXC 中備份。**

不進 git 的檔案在換裝置、重建環境或意外刪除時會永久遺失。KeePassXC 是這類檔案的唯一跨裝置備份來源。

---

## 備份對象

以下類型的檔案需要備份：

| 類型 | 常見範例 |
|------|---------|
| 環境變數檔 | `.env`、`.env.local`、`.env.production` |
| 服務帳號金鑰 | `*-sa-key.json`、`credentials.json` |
| 憑證檔 | `*.pem`、`*.p12`、`*.pfx` |
| 其他密鑰 | API token、webhook secret、SSH private key |

**不需要備份：**
- 虛擬環境（`.venv/`、`node_modules/`）— 可從 `requirements.txt` 或 `package.json` 重建
- 暫存或 build 產出 — 可重新產生

## KeePassXC Entry 結構

### 命名規則

**一個敏感檔案對應一個 Entry，標題直接反映檔案路徑：**

```
20260517_confluence/.env
20260517_confluence/middleware/.env
20260517_api/backend/.env
```

這樣的好處：
- 搜尋專案名稱即可找到所有相關 Entry
- 還原時路徑清楚，不會放錯位置
- 同一專案多個 `.env` 也能清楚區分

### 欄位填寫建議

| 欄位 | 填寫方式 |
|------|---------|
| **標題（Title）** | `<專案路徑>/<檔名>`，如 `20260517_confluence/.env` |
| **使用者名稱（Username）** | 留空，或填對應服務的帳號 Email |
| **密碼（Password）** | 留空（敏感值改放自訂欄位） |
| **備註（Notes）** | 完整檔案內容直接貼上（整份還原用） |
| **自訂欄位** | 個別重要值，設為保護欄位（方便單獨複製） |

### 自訂欄位 vs 備註欄

兩者**並存**，職責不同：

- **備註欄**：貼完整 `.env` 內容，換機器時整份複製貼上
- **自訂欄位**：列出會單獨用到的值（如 API token），方便日常複製不需要翻備註

### 備份方式選擇

| 方式 | 適用時機 | 操作位置 |
|------|---------|---------|
| **備註欄**（貼文字） | `.env` 等純文字檔 | Entry → 備註 |
| **附件**（上傳檔案） | 需要完整還原的二進位檔（如 JSON 金鑰） | Entry → 進階 → 附件 |

### 群組建議

在 KeePassXC 建立群組反映專案結構，避免多專案混雜：

```
KeePassXC
└── Projects/
    ├── 20260517_confluence/
    │   └── 20260517_confluence/.env
    └── 其他專案/
```

## 文件中的標準表格格式

每個專案的整合文件（如 `confluence-integration.md`）應包含以下區段，列出需要備份的檔案：

```markdown
## 🔐 需要備份的檔案（不進 git）

以下檔案含有密鑰或敏感設定，需手動備份至 KeePassXC：

| 檔案 | 內容 | KeePassXC 備份方式 |
|------|------|-------------------|
| `.env` | `VAR_A`、`VAR_B`、`VAR_C` | 備註欄（直接貼文字）|
| `sa-key.json` | GCP Service Account 金鑰 | 附件（還原時需要完整檔案） |

KeePassXC Entry Title 建議：`<專案名稱>/<檔名>`，例如 `smart_kanban/.env`。
```

**只列目前實際存在的檔案。** 已在 `.gitignore` 排除但尚未建立的檔案，以附註說明「建立後需備份」，不放進主表格。

## AI 行為協定

### 建立或修改 .gitignore 時

AI 應主動識別新增的排除項目是否屬於敏感檔案，若是，應：

1. 確認該檔案目前是否已存在（`ls` 或 `find`）：
   - **已存在** → 提醒使用者立即備份，並在整合文件備份表格補上對應項目
   - **尚未存在** → 在整合文件備份表格以附註標記「建立後需備份」，不放進主表格
2. 說明對應的 KeePassXC Entry Title 建議

### 新增敏感檔案時

當 AI 協助建立 `.env`、金鑰檔或其他敏感檔案時，應：

1. 確認該檔案已加入 `.gitignore`
2. 在專案整合文件的「🔐 需要備份的檔案」區段補上對應項目
3. 提示使用者同步更新 KeePassXC 備份

### 整理或審查備份清單時

當使用者要求整理、更新或確認備份清單時，AI 必須：

1. 列出 `.gitignore` 中排除的敏感檔案候選（`.env*`、`*-key.json`、`*.pem` 等）
2. 實際確認哪些檔案目前存在（執行 `ls` 或 `find`，不可憑印象判斷）
3. **只將目前已存在的檔案**列入備份主表格
4. 已在 `.gitignore` 但尚未建立的檔案，以附註說明「建立後需補備份」
5. 更新整合文件的「🔐 需要備份的檔案」區段

### 敏感檔案內容異動時

以下情境代表備份內容已過時，AI 應提醒使用者更新 KeePassXC：

- 密碼輪替或重設
- 新增或移除 env key
- AES key、API token、secret 異動
- 重新部署導致 secret 重新產生（如 `PLANKA_SECRET_KEY`、`POSTGRES_PASSWORD`）

### 禁止的行為

```
# 建立敏感檔案後沒有提醒備份                      ← 禁止
# 在整合文件中遺漏「需要備份的檔案」區段            ← 禁止
# 將敏感檔案加入 git 版本控制                      ← 禁止
# 整理備份清單前沒有實際確認哪些檔案目前存在        ← 禁止
# 備份清單列入目前不存在的檔案                      ← 禁止
# 敏感檔案內容異動後沒有提醒更新 KeePassXC 備份     ← 禁止
```

## 套用流程

**在新專案導入：**

1. 將此檔案放在專案 `specs/` 目錄，命名為 `specs/keepass-spec.md`
2. 在專案整合文件加入「🔐 需要備份的檔案」區段
3. 在 `CLAUDE.md`（或對應 AI 工具的設定檔）加入：

```
請遵照 specs/keepass-spec.md 的規範，建立敏感檔案時提醒備份至 KeePassXC。
```

| 其他工具 | 對話開頭直接引用本檔案內容 |
