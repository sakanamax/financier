# AI 協作開發規範

適用情境：有環境變數、部署流程、建置驗證或操作紀錄需求，且由 AI 工具協作的軟體專案。

若專案沒有部署、建置或外部環境設定，本規範仍可作為檢查清單使用，但可按實際情況略過不適用的章節，並在回報中說明略過原因。

---

## 核心原則

**動手前先讀既有規格；動手後把結果記在對的地方。**

---

## 規則一：寫 env 讀取程式碼前，必須先確認 env 檔案內容

### 問題來源

2026-05-18 建立 `middleware/config.py` 時，直接自行命名 env 變數
（`MIS_API_URL`、`MIS_AES_KEY`、`MIS_EMP_ID`），
沒有先讀 `.env.container`，導致三個變數名稱與已存在的 env 檔對不上，
執行階段才發現並修正。

### 規則

任何新增或修改讀取 env 的程式碼前，必須先確認專案中已存在的 env key。

優先檢查以下檔案；不存在的檔案可略過：

- `.env`
- `.env.local`
- `.env.container`
- `.env.container.example`
- `.env.vm.example`
- `.env.example`
- `docker-compose.yml`
- `docker-compose.yaml`

檢查 env 檔案時，使用只顯示 key、隱藏 value 的方式，例如：

```bash
grep -v '^\s*#' .env.container | grep -v '^\s*$' | sed 's/=.*/=***/'
```

若 env key 來自 `docker-compose.yml`、Kubernetes Secret、CI/CD 設定或雲端部署設定，必須同時確認來源檔案或設定頁面中實際使用的名稱。

確認所有 key 名稱與用途後，再動手寫 config。

### 禁止的行為

```
# 直接照自己的命名習慣定義 env 變數名稱               ← 禁止
# 假設 env key 名稱而不查證                           ← 禁止
# 寫完 config 才去核對 env 檔案                       ← 禁止
```

## 規則二：完成部署或建置後，必須更新部署文件

### 兩份文件的分工

| 文件 | 記什麼 | 問法 |
|---|---|---|
| `docs/deployment/validation_log.md` | **這次發生了什麼**：操作、結果、問題、處置 | 「我今天做了什麼、遇到什麼、怎麼解決的？」 |
| `docs/deployment/runbook.md` | **下次怎麼重複做**：可逐步執行的操作步驟 | 「六個月後另一個人要怎麼重現這件事？」 |

### 觸發條件

以下任一發生時，AI 必須同步更新：

| 情境 | validation_log | runbook |
|---|---|---|
| 新建程式碼骨架並驗證 | ✅ 記錄結果與問題 | ✅ 補驗證步驟 |
| 部署腳本執行成功 | ✅ 記錄驗證結果 | ✅ 補或更新操作步驟 |
| 遇到問題並修正 | ✅ 記錄問題與處置 | 若有 workaround 需補注意事項 |
| 新增操作方式（指令、endpoint） | 不需要 | ✅ 補操作步驟 |
| 變更設計決策 | ✅ 記錄決策內容 | 若影響操作流程則更新 |

**沒有對應更新不是跳過的理由，而是補寫的信號。**

### 豁免條件

以下情況可不更新部署文件，但回報時必須說明原因：

- 純文件文字修訂，未改變操作流程、部署方式或驗證結果。
- 純內部重構，未新增 env、指令、endpoint、部署腳本或外部操作步驟。
- 僅修正註解、格式、命名或測試資料，且不影響建置、部署或驗證方式。

若不確定是否豁免，預設更新 `validation_log`；只有可重複操作方式改變時才更新 `runbook`。

### validation_log 格式

每筆紀錄使用固定格式，不得省略欄位：

```
## YYYY-MM-DD：<本次操作標題>

環境：
目的：
操作：（bullet list）
結果：（bullet list）
問題：（若無寫「無」）
處置：（若無寫「無」）
後續：
```

### runbook 格式

- 以有意義的階段或功能區塊分節（例如：環境準備、服務部署、驗證步驟），讓讀者可以按需跳讀。
- 每個步驟附上可複製貼上的指令與預期結果。
- 若操作有前置條件或注意事項，以「注意：」開頭列出。
- 待補的步驟明確標示「待 X 完成後補齊」，不留空白節。

### 禁止的行為

```
# 做完驗證但沒有更新 validation_log          ← 禁止
# 新增了可重複步驟但沒有更新 runbook          ← 禁止
# 只更新其中一份就聲稱完成                    ← 禁止
# runbook 寫了步驟但沒有預期結果              ← 禁止
# validation_log 省略「問題」或「處置」欄位   ← 禁止（無問題寫「無」）
```

## AI 行為協定

### 每次操作前（規則一）

進行以下操作前，AI 必須先列出 env key 清單：

- 新增讀取 env 的模組（`config.py`、`settings.py` 等）
- 修改現有 env 變數名稱
- 建立部署腳本中讀取 env 的部分

### 每次操作後（規則二）

完成以下類型的工作後，AI 必須依序：

1. **識別**：本次工作屬於哪個觸發情境？
2. **更新 validation_log**：用固定格式記錄結果與問題。
3. **更新 runbook**：補上可重複執行的操作步驟。
4. **回報**：附上兩份文件的更新摘要。

回報格式：

```
觸發情境：
validation_log：
runbook：
驗證結果：
未更新原因：（若不適用則寫「不適用」）
```

## 套用流程

**在新專案導入：**

1. 將此檔案複製至專案根目錄，命名為 `dev-spec.md`
2. 建立 `docs/deployment/validation_log.md`（空白或含格式說明）
3. 建立 `docs/deployment/runbook.md`（以 Phase 分節）
4. 在 `CLAUDE.md`（或對應 AI 工具的設定檔）加入：

```
請遵照 dev-spec.md 的開發撰碼規範。
```

**與 AI 工具協作時告知：**

> 「請遵照 `dev-spec.md` 的規範：動手前先確認既有規格；動手後更新 validation_log 與 runbook。」
